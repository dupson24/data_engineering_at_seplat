    Introduction

    Cofense PhishMe® enables you to access information about your organization’s program in several ways. You can always refer to the excellent charts, tables, and reports available through the Cofense PhishMe interface. However, you might also want to use the Cofense PhishMe API to extract the organization’s data for use in your tools outside of the PhishMe interface. Cofense PhishMe operators can configure the API to pull data on demand or automatically upon campaign completion.

    We offer language bindings in Shell, Ruby, and Python. You can view code examples in the shaded code panel located on the right side of the screen. Use the tabs at the top of the code panel to display your desired programming language.

    Our documentation will provide information on our API (v2) endpoints. Endpoints use the same token-based authentication.

    You can only see your API token when you generate it, so make a copy and keep it safe. If you misplace your API token, you must create a new one.

    Who Can Use This Feature?

    All Cofense PhishMe operators, except operators with the anonymous role, can generate an API token. The Cofense PhishMe API tokens are available on a per operator basis. If you have permission to use this feature, you can manage API tokens from your account profile.

    Note Regarding Deprecation of v1 API

    Cofense has announced end-of-support and end-of-life for the v1 API as of August 31, 2025. Plan to transition all scripts and automation tools to our v2 API before then to avoid disruption of services and workflows.

    API URL

    In each API request, you must pass the API URL. The following table identifies the correct API URL to include in your requests based on the URL you use to log in to Cofense PhishMe and the API version indicated for an endpoint.

    API Version Cofense PhishMe Login URL API URL
    v2 https://login.phishme.co.uk/ https://login.phishme.co.uk/api/v2/

    Authentication

    API v2

    To authorize with our v2 API, use this code:

    # With shell, you can just pass the correct header with each request
curl -s "https://login.phishme.co.uk/api/v2/scenarios" \
     -H 'Authorization: Token token=2d61b1f466f4e79f5687fee5b6dddd6d'

# Our API endpoint.
url = "https://login.phishme.co.uk/api/v2/scenarios"

# We need to add our API token to the request to authorize it.
headers = {'Authorization': 'Token token="2d61b1f466f4e79f5687fee5b6dddd6x"'}

    Make sure to replace the sample API key 2d61b1f466f4e79f5687fee5b6dddd6x with your API key.

    After you generate a token, you must insert the token into the authorization request header of each API request. The correct way to set the token in the header depends on the programming language or tool you use to access the API. You can use the same API token in both APIs that PhishMe offers.

    Cofense PhishMe uses API keys to allow access to the API. You can generate a new token on your Account Profile page.

    Cofense PhishMe expects the API key to be in all API requests to the server. An authorization request header looks like the following:

    Authorization: Token token="yourToken"

    You must replace yourToken with your personal API key.

    Token Permissions

    Cofense PhishMe offers two types of tokens for accessing the API: read-only, and read-write.

    Tokens of type read-only grant general access to any API endpoint that is not updating a record (GET requests). POST, PATCH, and DELETE endpoints, are restricted in general, with some exceptions. POST endpoints that enqueue an export file, as used in the repeat_clickers_exports and scenario_exports endpoints, are not restricted, as no underlying data is being modified.

    Tokens of type read-write grant access to all API endpoints accessible to read-only tokens, as well as POST, PATCH, and DELETE endpoints allowing for the creation, modification, and destruction of records. Take the same care when accessing these endpoints as you would when creating, modifying, or deleting data in the application's user interface.
